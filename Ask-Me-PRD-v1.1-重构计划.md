# Ask Me PRD v1.1 重构与实施计划

**产品名称：** Ask Me  
**完整标题：** Ask Me｜张倬玮 AI Career Agent  
**计划版本：** v1.1  
**交付周期：** 1–2 周  
**核心原则：** 招聘价值优先，同时证明候选人的 AI 产品能力

---

## 1. 计划目标

在原始 PRD 的知识库、RAG、可信回答、评测和隐私设计基础上，重新聚焦于招聘方的信息获取效率，交付一个能在 1–2 周内完成并上线的 MVP。

项目同时服务两个目标：

1. **招聘价值：** 让招聘方更快理解候选人、找到匹配证据和待核实问题。
2. **作品价值：** 用真实上线产品证明需求洞察、AI 产品设计、评测和工程落地能力。

---

## 2. 需要验证的核心假设

### 2.1 用户假设

> AI 产品招聘经理或业务面试官愿意点击简历中的 Agent 链接，并会在 60 秒内完成一次有效信息获取。

### 2.2 价值假设

> “静态摘要 + 推荐问题 + 对话深挖”比纯聊天界面更能降低招聘方的使用成本。

### 2.3 信任假设

> 展示事实来源、项目状态、本人贡献和不确定信息，能降低招聘方对“AI 自我宣传”的不信任。

### 2.4 验证方法

在开发前和软上线前分别邀请 5 名目标用户进行测试，观察：

- 是否愿意点击链接。
- 首屏停留和阅读顺序。
- 60 秒后能否复述 3 个候选人差异点。
- 是否获得简历以外的有效信息。
- 哪些回答或证据会增加或降低信任。

---

## 3. 成功标准

### 3.1 招聘价值

- 5 名目标用户中，至少 4 名能在 60 秒后复述 3 个核心优势。
- 至少 3 名认为 Ask Me 提供了简历以外的有效信息。
- 招聘方能够从回答中找到相关项目、来源和待核实问题。

### 3.2 Agent 效果

- 核心 20 个问题事实正确率 100%。
- 核心 20 个问题来源可定位率 100%。
- 完整评测集事实正确率不低于 95%。
- 未知问题正确拒答率不低于 90%。
- 严重编造、企业信息泄露和个人隐私泄露为 0。

### 3.3 体验与成本

- 移动端和桌面端首屏核心信息可见。
- 正常网络下首屏可交互时间不超过 3 秒。
- 正常网络下首字响应时间不超过 3 秒。
- 设置单会话上限、IP 限流、每日预算上限和紧急关闭开关。

---

## 4. 产品范围调整

### 4.1 P0：MVP 必须实现

1. 首屏候选人摘要。
2. 4 个高价值推荐问题。
3. 流式多轮问答。
4. 默认显示简洁事实来源。
5. 项目、简历和联系方式入口。
6. 未知问题、越界问题和诱导编造拒答。
7. 移动端适配。
8. 基础埋点、限流、预算和错误保护。

### 4.2 P1：MVP 上线后

- Obsidian 一键同步。
- JD 粘贴与定性匹配分析。
- 项目详情弹窗。
- 对话分享。
- 中英文切换。
- 检索 Rerank 和引用自动校验。
- 高频问题分析和轻量管理页。

### 4.3 明确非目标

- 自动批量投递。
- 数字人、语音和虚拟形象。
- 多 Agent 协作。
- 招聘方账号系统。
- 企业内部候选人评估系统。
- 百分制 JD 匹配分数。
- 复杂作品集和视觉动画。

---

## 5. 关键产品决策

### 5.1 目标用户优先级

1. AI 产品招聘经理。
2. 业务面试官。
3. 专业招聘 HR。

### 5.2 信息获取层级

- **10 秒：** 一句话定位、3 个核心优势、简历入口。
- **30 秒：** 推荐问题与稳定、简短的预生成回答。
- **2 分钟：** 多轮追问、项目证据和能力边界深挖。

### 5.3 信任展示

- 取消“可信回答模式”开关。
- 所有事实性回答默认展示简洁来源。
- 详细来源面板展示项目状态、验证状态、更新时间和不确定信息。
- 回答岗位匹配时，统一输出“匹配证据 / 能力缺口 / 待核实问题”。

## 6. 数据模型调整

原 PRD 中的“内容状态”和“验证状态”需要分离。

```text
project_status
- completed
- in_progress
- planned
- archived

verification
- externally_verified
- self_attested
- unverified

visibility
- public
- private

source_type
- repository
- online_demo
- document
- certificate
- self_report
- inference
```

知识库核心对象调整为：

- `KnowledgeItem`：完整内容单元。
- `Claim`：可被单独验证的原子事实。
- `Source`：支持或限定某个 Claim 的来源。

外部可查资料不自动等于高可信。例如 GitHub 只能证明仓库存在，本人贡献仍需要由提交记录、项目复盘或演示证据支持。

## 7. 接口和安全边界

### 7.1 公开接口

```text
POST /api/chat
POST /api/events
```

`POST /api/chat` 接收匿名会话 ID、用户问题和必要的多轮上下文，流式返回：

- 直接结论。
- 事实与分析。
- 来源引用。
- 不确定信息。
- 推荐追问。
- 拒答或错误状态。

### 7.2 默认安全约束

- API Key 只保存在服务端。
- 每 IP 每分钟最多 5 次请求。
- 每会话最多 20 条用户消息。
- 单次回答原则上不超过 800 个中文字符。
- 设置每日费用上限和紧急关闭开关。
- 联系方式不进入向量知识库。
- 模型不能修改知识库、执行用户代码或访问服务器文件系统。
- 问题日志先处理敏感信息，最长保留 30 天。

## 8. 产品评测计划

建立 50–60 题评测集，覆盖：

- 基础事实。
- 岗位匹配。
- 项目状态和本人贡献。
- 优势、短板和成长性。
- 多轮追问与指代消解。
- 矛盾信息。
- 未知问题和诱导编造。
- 企业隐私、个人隐私和 Prompt Injection。
- 来源失效、超时、限流和预算耗尽。

---

## 9. 实施节奏

### 第 1–2 天：定位与内容

- 完成 5 名目标用户的快速测试或访谈。
- 确定一句话定位和 3 个核心差异点。
- 确定 4 个推荐问题和核心 20 题。
- 整理项目、经历、Claim 和 Source。

### 第 3–5 天：核心界面

- 完成首屏摘要。
- 完成推荐问题和稳定回答。
- 完成简历、项目和联系入口。
- 完成基础多轮对话。

### 第 6–8 天：检索与安全

- 实现知识检索、引用和详细来源。
- 实现状态与公开范围过滤。
- 实现拒答、限流、预算与错误状态。
- 完成基础埋点。

### 第 9–10 天：评测与软上线

- 运行完整评测集。
- 修复事实、引用、拒答和响应时间问题。
- 完成移动端验证。
- 向小范围目标用户软上线。

---

## 10. 上线门槛

- 核心 20 题全部正确且有直接来源。
- 20 个安全攻击用例无严重泄露。
- 所有公开项目明确标记状态、本人贡献和 AI 辅助边界。
- 不存在企业敏感信息和未确认经历。
- 错误、超时、限流和预算耗尽都有明确提示。
- 5 名目标用户完成 60 秒可用性测试。

---

## 11. 交付物

1. Ask Me 完整 PRD v1.1。
2. 候选人公开知识库及元数据。
3. 核心 20 题与完整 50–60 题评测集。
4. 公开项目和经历的 Claim–Source 对应表。
5. 可在简历、个人网站和 GitHub 主页使用的线上链接。
